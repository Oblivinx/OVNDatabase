OvnDB
v4.0

Database NoSQL embedded berkinerja tinggi untuk Node.js. API expressive berbasis dokumen JSON, berjalan langsung di proses kamu tanpa server. Murni TypeScript. Zero dependency eksternal.

⚡ ~20K insertOne/s 🔍 B+ Tree On-disk 🔒 AES-256-GCM 🧠 MVCC Snapshot Isolation 🌸 Bloom Filter 📝 WAL Crash-safe 🔎 Full-text Search 💾 Multi-segment Storage
💡
Filosofi inti: Kekuatan embedded storage yang bekerja langsung di proses kamu, dipadukan dengan model dokumen JSON yang fleksibel dan query operator yang ekspresif — semuanya tanpa dependency eksternal.

🎯Kenapa OvnDB?

OvnDB dirancang untuk use case yang membutuhkan database powerful yang berjalan langsung di proses aplikasi — tanpa overhead infrastruktur, tanpa dependency eksternal.

🚀
Embedded — No Server
Berjalan langsung di proses Node.js kamu. Tidak perlu install MongoDB, tidak perlu Docker, tidak perlu konfigurasi jaringan. Buka file, langsung query.
🍃
API MongoDB-Compatible
Kalau kamu sudah familiar dengan MongoDB — insertOne, find, updateOne, aggregate — kamu langsung bisa pakai OvnDB tanpa belajar ulang.
📈
Skala ke Triliunan Record
Multi-segment storage (tiap segment max 512 MB, auto-expand). Tidak ada batas jumlah record — data disimpan di disk, bukan RAM.
🔐
Enkripsi Bawaan
AES-256-GCM per-record dengan authenticated encryption. Tidak perlu library pihak ketiga. Cocok untuk data sensitif seperti PII, data kesehatan, dll.
🛡️
Crash-safe by Design
Write-Ahead Log (WAL) dengan group commit — setiap write di-persist ke WAL sebelum ke storage. Kalau listrik mati, data tidak corrupt.
0️⃣
Zero External Dependencies
Tidak ada npm install yang bisa bermasalah. Tidak ada native bindings yang perlu di-compile. Tidak ada supply chain attack vector. Pure JavaScript.

🧬Filosofi & Inspirasi

OvnDB lahir dari dua dunia yang berbeda — keduanya menjadi fondasi desain, bukan saingan. Setiap keputusan arsitektur di OvnDB bisa ditelusuri kembali ke inspirasi ini.

📦
Dari SQLite: Embedded & Local
Filosofi "database sebagai file" — tidak ada daemon, tidak ada server, tidak ada koneksi jaringan. Cukup buka path, langsung kerja. Prinsip ini diterapkan di OvnDB: OvnDB.open('./data') adalah satu-satunya pintu masuk. Data tersimpan di disk lokal dalam format binary terstruktur dengan page-based storage, persis seperti paradigma embedded database.
🍃
Dari MongoDB: API & Model Dokumen
Filosofi "schema-less, expressive query" — dokumen JSON fleksibel, query operator yang kaya ($gte, $in, $elemMatch…), aggregation pipeline berlapis, dan Change Stream. OvnDB mengadopsi keseluruhan model ini sehingga siapapun yang familiar dengan MongoDB bisa langsung produktif tanpa kurva belajar baru.

OvnDB bukan sekadar menggabungkan — ia menambahkan lapisan yang tidak dimiliki keduanya secara bawaan:

🌳B+ Tree — Kenapa Dipilih?

OvnDB menggunakan On-disk Paged B+ Tree untuk primary index dan secondary index. Ini adalah pilihan teknis yang disengaja dengan alasan kuat.

Apa itu B+ Tree?

B+ Tree adalah struktur data tree yang self-balancing. Semua data tersimpan di leaf nodes yang saling terhubung sebagai linked list. Internal nodes hanya menyimpan keys sebagai "router" untuk navigasi. Ini berbeda dari B-Tree biasa di mana data bisa ada di internal nodes.

Alasan Memilih B+ Tree

🏃
O(log N) untuk Semua Operasi
Insert, delete, get — semua O(log N). Dengan T=128 (max 255 keys per node), tree dengan 1 juta record hanya memiliki ~3-4 level. Traversal sangat cepat.
📊
Range Query Sangat Efisien
Leaf nodes terhubung sebagai doubly-linked list. Range query ($gte, $lte) cukup cari leaf pertama lalu scan ke kanan — tidak perlu backtrack ke atas.
💾
On-disk dengan Buffer Pool
Pages di-cache di RAM (256 MB buffer pool LRU). Hanya pages yang diperlukan yang di-load ke memori. Tree bisa jauh lebih besar dari RAM yang tersedia.
🔢
Ordered — Cursor Pagination
Data selalu terurut. Cursor pagination (after) jauh lebih efisien dari skip karena langsung jump ke posisi di tree, bukan scan dari awal.

Implementasi di OvnDB

Lihat file src/core/index/btree-paged.ts. Parameter kunci:

// src/core/index/btree-paged.ts
const T        = 128;          // minimum degree
const MAX_KEYS = 2 * T - 1;  // = 255 keys per node

// Tiap pointer ke record menyimpan:
// segmentId (4B) + offset (8B) + totalSize (4B) + dataSize (4B) + txId (8B)
const POINTER_SIZE = 4 + 8 + 4 + 4 + 8;  // = 28 bytes

btree-paged.ts
🔬
Kenapa bukan Hash Map? Hash map lebih cepat untuk point lookup (O(1)) tapi tidak bisa range query ($gte, $lte, sort). B+ Tree mendukung semua query patterns yang dibutuhkan database dokumentasi.
🔬
Kenapa bukan LSM Tree (seperti LevelDB/RocksDB)? LSM Tree lebih cepat untuk write-heavy workload tapi lebih kompleks — butuh compaction background yang bisa menyebabkan write stalls. B+ Tree memberikan performa yang lebih predictable untuk mixed read/write.

Cursor Pagination dengan B+ Tree

// Lebih efisien dari skip — langsung jump ke posisi di tree
const page1 = await col.find({}, { limit: 20, sort: { _id: 1 } });
const page2 = await col.find({}, {
  limit: 20,
  sort: { _id: 1 },
  after: page1.at(-1)?._id  // cursor dari item terakhir page sebelumnya
});

💽Storage Engine

OvnDB menggunakan multi-segment storage — data collection disimpan di beberapa file segment, masing-masing maksimal 512 MB.

Arsitektur Storage

StorageEngine
├── SegmentManager ← petabyte-scale multi-file storage
│ ├── BloomFilter v4.0 ← per-segment, FNV-1a double-hash, 1% FPR
│ └── .seg-0000.ovn, .seg-0001.ovn ...
├── PagedBPlusTree ← on-disk index, 256 MB buffer pool
│ └── PageManager ← LRU buffer pool → disk
├── WAL (group commit) ← crash recovery, auto-rotate 256 MB
├── LRUCache<string, Buffer> ← byte-aware doc cache, 256 MB default
└── MVCCManager ← Snapshot Isolation + Savepoints

Multi-Segment Storage

Setiap collection memiliki beberapa segment file. Ketika segment mencapai 512 MB, segment baru dibuat otomatis. Ini memungkinkan collection tidak terbatas ukurannya.

Aspek Detail
Max ukuran per segment 512 MB (konfigurabel)
Jumlah segment Tidak terbatas — auto-expand
Format header segment 4 bytes magic OVN\x01
Compaction Background compaction dengan rasio threshold 60%
Parallel compaction Segment-level concurrency

LRU Cache — Byte-Aware

Cache diukur dalam bytes, bukan jumlah entry. Default 256 MB. Ini lebih akurat karena dokumen bisa punya ukuran yang sangat berbeda.

const db = await OvnDB.open('./data', {
  cacheBytes: 512 * 1024 * 1024,  // 512 MB cache
  fileLock:   true,                // cegah multi-process corruption
  mkdirp:     true,                // auto-create directory
  gracefulShutdown: true,          // auto-close pada SIGTERM/SIGINT
});

📝WAL & Crash Recovery

Write-Ahead Log (WAL) memastikan data tidak pernah corrupt meskipun proses crash di tengah operasi.

Bagaimana WAL Bekerja

Setiap write pertama ditulis ke WAL file (.wal), baru kemudian ke segment storage. Saat crash, WAL di-replay ulang untuk memulihkan state terakhir.

⚡
Group Commit
Bukan 1 fsync per operasi (seperti v1). WAL mengumpulkan batch operasi dalam window waktu pendek (~1ms) lalu 1 fdatasync untuk semua. ~10× throughput lebih tinggi.
🔄
Auto-rotation
WAL file di-rotate ketika mencapai 256 MB. File lama di-rename ke .wal.bak, file baru dibuat. Crash-safe via atomic rename (.wal.new → .wal).
🛑
TX_ABORT Tracking
WAL melacak operasi yang di-abort. Saat replay, operasi dari transaksi yang di-abort TIDAK di-replay. Fix bug corruption yang ada di v2.x.
✅
CRC32 Checksum
Setiap entry WAL punya CRC32 checksum. Saat replay, entry yang corrupt (partial write karena crash) dideteksi dan di-skip.
⚠️
WAL otomatis dibersihkan setelah flush() berhasil. WAL file kecil berarti database sering di-checkpoint — itu tanda yang baik.

Savepoints (v4.0)

Rollback parsial tanpa membatalkan seluruh transaksi.

const tx = db.beginTransaction();

tx.insert(col, { _id: 'a', balance: 100 });
tx.insert(col, { _id: 'b', balance: 200 });

// Buat savepoint sebelum operasi berisiko
tx.savepoint('before-risky');

tx.insert(col, { _id: 'c', balance: 999 });  // operasi berisiko

// Rollback hanya sampai savepoint — 'a' dan 'b' tetap
await tx.rollbackTo('before-risky');

// Commit — hanya 'a' dan 'b' yang masuk
await tx.commit();

🌸Bloom Filter (v4.0)

Probabilistic data structure yang digunakan untuk skip disk I/O ketika key pasti tidak ada di suatu segment. Menghemat hingga 80%+ disk reads untuk miss queries pada dataset besar.

🎯
False Positive Rate ~1%
Bloom filter mungkin bilang "ada" padahal tidak (false positive), tapi tidak pernah bilang "tidak ada" padahal ada (false negative). Safe untuk skip optimization.
🔢
7 Hash Functions (k=7)
Jumlah hash functions optimal untuk FPR ~1%. Implementasi FNV-1a dengan double hashing — fast, no crypto overhead.
// Cara kerja Bloom Filter di OvnDB:
// 1. Saat insert record ke segment, key di-add ke bloom filter segment itu
// 2. Saat findOne(key), cek bloom filter SEBELUM B+ Tree lookup
// 3. Jika bloom filter bilang "pasti tidak ada" → skip segment ini, coba segment lain
// 4. Jika "mungkin ada" → baru lakukan B+ Tree disk lookup

const bf = new BloomFilter(10_000);  // expected 10K items, FPR ~1%
bf.add('user:u1');
bf.test('user:u1');   // true  — mungkin ada → perlu B+ Tree check
bf.test('user:u9');   // false — pasti tidak ada → skip disk I/O ✓

🚀Quick Start

// Install
npm install ovndb

import { OvnDB } from 'ovndb';

// 1. Buka database (auto-create direktori)
const db    = await OvnDB.open('./data');
const users = await db.collection<User>('users');

// 2. Insert
const user = await users.insertOne({ name: 'Budi', phone: '628123', points: 0 });
// → { _id: 'abc123...', name: 'Budi', phone: '628123', points: 0 }

// 3. Find
const vips = await users.find(
  { points: { $gte: 1000 } },
  { sort: { points: -1 }, limit: 10 }
);

// 4. Update
await users.updateOne({ _id: user._id }, { $inc: { points: 50 } });

// 5. Delete
await users.deleteOne({ _id: user._id });

// 6. Tutup
await db.close();

Single-field Index

Index multi-field untuk query yang sering menggunakan kombinasi kondisi. Key compound diserialisasi sebagai val1\x00val2 (NUL separator).

📐Schema Validation

import { SchemaValidator, field } from 'ovndb';

const schema = new SchemaValidator({
  name:   field('string').required().minLength(2).maxLength(100).build(),
  phone:  field('string').required().pattern(/^628\d{8,12}$/).build(),
  points: field('number').min(0).max(1_000_000).default(0).build(),
  tags:   field('array').maxItems(20).items(field('string').build()).build(),
  role:   field('string').enum('admin', 'user', 'guest').build(),
});

// Soft validation (tidak throw)
const { ok, errors } = schema.validate(doc);

// Hard validation (throw ValidationError)
schema.validateOrThrow(doc);

// Apply default values
schema.applyDefaults(doc);
schema.ts
🛡️

🔒Enkripsi AES-256-GCM

import { OvnDB, cryptoFromPassphrase } from 'ovndb';

const db  = await OvnDB.open('./data');
const key = await cryptoFromPassphrase(process.env.DB_KEY!, './data');

// CollectionV2 — enkripsi per-record otomatis
const users = await db.collectionV2<User>('users', {
  crypto:  key,
  indexes: [{ field: 'phone', unique: true }],
});

// API identik dengan Collection biasa — transparan
const user = await users.findOne({ phone: '628123' });

// Key rotation — re-encrypt semua record dengan key baru

import { TTLIndex } from 'ovndb';

const sessions = await db.collection('sessions');

  field:         'expiresAt',

🔢Format Record On-disk

Setiap record di segment file disimpan dengan format berikut:

1 byte
status
0x01 = ACTIVE
0x00 = DELETED
8 bytes
txId
MVCC version
(BigInt uint64 LE)
4 bytes
dataLen
payload length
(uint32 LE)
N bytes
data
JSON payload
(atau ciphertext jika enkripsi aktif)
4 bytes
CRC32
checksum semua field sebelumnya

Total overhead per record: 1 + 8 + 4 + 4 = 17 bytes (di luar payload data).

📁Struktur File

data/
└── users/ ← satu direktori per collection
    ├── users.manifest.json ← daftar segment + metadata (SHA-256 / HMAC)
    ├── users.seg-0000.ovn ← segment 0 (max 512 MB)
    ├── users.seg-0001.ovn ← segment 1 (auto-created)
    ├── users.ovni ← on-disk B+ Tree index (page file)
    ├── users.wal ← Write-Ahead Log (dibersihkan setelah flush)
    ├── users.idx.city ← secondary index field 'city'
    ├── users.idx.city,active ← compound index ['city','active'] (v4.0)
    ├── users.fts.name.json ← FTS inverted index field 'name' (v4.0)
    └── .salt ← PBKDF2 salt (jika enkripsi aktif)
